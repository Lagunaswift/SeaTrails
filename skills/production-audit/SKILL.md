# Production Audit (orchestrator)

This is the master audit. Its job is not to re-do the specialist skills' work but to **coordinate** them: run every relevant lens over the app, then merge, verify, and consolidate their findings into one prioritised report instead of a dozen disconnected ones. A single lens answers one question (is it secure? will it scale?); this answers the only question that matters before shipping, **is this app actually ready, across every dimension, and what must be fixed first?**

It wraps rather than replaces the specialists. `code-audit` remains the deep security/correctness engine and is still run directly when security is all that's wanted; this skill calls it as one pass among many and adds the cross-lens consolidation on top.

## The cardinal principle

**The value is in the consolidation, not the concatenation.** Running fifteen lenses and stapling their outputs together produces a sprawling, duplicated, uncalibrated mess that buries the few things that actually matter. The orchestrator earns its place by doing what no single lens can: deduplicating the same issue seen through multiple lenses, resolving severity disagreements, dropping false positives through adversarial verification, and ordering everything by what to fix first. A consolidated report of 40 real, ranked, verified findings is worth more than 142 raw ones.

## Always report mode

Default to report mode: assess and report, change no code. Fixing is a separate, deliberate phase done under the audit discipline (establish a test net first, separate structural from behavioural changes, verify each step, governed by `testing-strategy` and `refactoring`). Offer to move to fix mode only after the report is delivered and the user chooses what to address, in the recommended order.

## The lens registry

`references/lens-registry.md` is the coordination map. It lists every lens with: the backing skill file, the passes it runs, what it owns (non-overlapping with other lenses), and supporting references the subagent should read. The orchestrator **must** consult this registry when spawning subagents — a subagent that runs a lens without reading its backing skill produces ad-hoc findings instead of structured ones, and the quality difference is the entire point of having skills.

## What makes this drop-in and foolproof

This skill is built to drop into *any* repository and produce a trustworthy compiled report — not to assume one stack, and not to let the agent merely *claim* rigour. An honest boundary: the audit *process* (detect the stack, select lenses, run each, append findings) is still carried out by an agent following these stages — that part rests on discipline. What is **machine-enforced** is the *output's integrity*: the harness gates the final report and mechanically rejects the failure modes that discipline alone has let through before (lost findings, unverified criticals, laundered severities, skipped lenses). So the claim is precise — not "the agent can't make a mistake," but "a mistake that corrupts the report's integrity cannot ship past the harness." Four pieces of machinery enforce that, and they are not optional:

- **Stack detection (`references/stack-adaptation.md`).** Stage 0 profiles the repo (language, framework, datastore, hosting, services) so lenses adapt instead of emitting Next.js/Firebase noise in a Python or Go repo. The lens examples are illustrations of universal questions; this translates them to the stack in front of you and adds stack-specific checks (e.g. SQL injection on a SQL stack).
- **The canonical finding schema (`references/finding-schema.md`).** Every lens emits findings in one machine-checkable shape with a stable id. "Standard format" means this schema, nothing looser.
- **The ledger and reconciliation (`references/ledger-and-reconciliation.md`).** Every raw finding is appended to `raw-findings.jsonl` as it's discovered and ends in exactly one recorded state — reported, merged, or dropped. This is the structural fix for the worst failure this audit has had: findings summarised away between discovery and report, taking real criticals with them.
- **The integrity harness (`scripts/audit-check.mjs`).** A zero-dependency Node script that hard-fails the audit if any critical/high is unverified, if any raw finding was lost, if the counts don't reconcile, if the schema is violated, if coverage is unmeasured or incomplete without acknowledgement, if gating findings lack a remediation order, if cited files don't exist in the repo (`--repo` flag), or if report prose contains AI-slop phrases. It is the exit gate: the report is not delivered until the harness exits zero. Run it with `node .claude/skills/production-audit/scripts/audit-check.mjs <audit-dir> [--repo <path>]`.

Coverage is made measurable too (`references/coverage-matrix.md`): the report states which lenses examined which areas and what fraction of source files were seen, so "all issues" has a denominator and gaps are visible rather than silent. Coverage is now a hard gate, not a warning — incomplete coverage without `scope.partial=true` blocks the audit.

## The process

The audit runs in six stages (Stage 0 plus Stages 1–5). The references detail each. Stage 0 and the four pieces of machinery above are what let it produce, in an unfamiliar repo, a report whose rigour is checked, not merely asserted.

**There is no single command that runs the whole audit.** The audit is executed by an agent (you) working through Stages 0–5; the scripts under `scripts/` do not perform the audit — `audit-check.mjs` *gates* the result and `render-report.mjs` *renders* it from the gated data. Running `audit-check.mjs` against an empty or absent audit-dir will not produce an audit; it will report that there is nothing to check. So the drop-in model is: drop the `.claude/skills/` tree into the target repo, invoke this skill, and the agent walks the stages writing `raw-findings.jsonl` and `report.json`; the two scripts then gate and render the deliverable.

## Budget the run before starting: a full sweep rarely fits one session
A complete audit across all applicable lenses is large (a real run can produce 50+ raw findings before consolidation), and it will often not finish in a single session. The failure mode to avoid is running lenses in arbitrary order until a limit cuts the audit off mid-sweep, leaving a report that covers whatever happened to run first (usually security) and silently omits the rest, a "production audit" that is really a third of one. Plan for this up front:
- **Estimate the scope.** Many applicable lenses on a substantial codebase will not complete in one pass. Decide at the start whether this is a one-session run or a staged one.
- **If staging, split by priority and tell the user.** Run the highest-stakes lenses first (security, AI security, correctness, then scaling, ops, data-privacy), deliver that as an explicit *partial* report, and state clearly which lenses remain and that a follow-up run is needed to finish them. Do not present a partial run as a complete audit.
- **Make partial-run honesty a feature, not an accident.** Whenever the audit does not cover every applicable lens, whether by plan or by interruption, the method line and the "what could not be assessed" section must name every lens that did not run and state that a follow-up is required (see `references/report-format.md`). A partial audit that says so plainly is useful; a partial audit that looks complete is dangerous, because the reader trusts coverage that never happened.
- **Resuming.** A follow-up run re-states scope, lists which lenses already ran (and where that report is), and runs the remainder, so the two reports together cover the whole set without re-doing work.

The goal is that an interrupted or staged audit is always clearly labelled as partial with its gaps named, never mistaken for a finished sweep.

### Stage 0: Stack detection
Before selecting lenses, profile the repository: language(s), framework, datastore family (SQL / document / none), rendering model (SSR / SPA / static / none), hosting, third-party services, **and the data classes it handles** (`stack_profile.data_classes` — special-category/health, financial, children's, government-id, …). This drives which lenses apply, how each lens's checks are phrased, and — crucially — whether a **legal data-protection duty** is triggered: a regulated data class makes the compliance pass mandatory regardless of any stated goal (see Stage 1). It is what lets the audit drop into any repo without emitting findings tied to a stack the repo doesn't use, or missing checks (like SQL injection, or a DPIA obligation) the lens examples didn't foreground. Record the profile in `report.json`. `references/stack-adaptation.md` covers the detection signals, the data-class table, and the cross-stack translation table.

### Stage 1: Scope and lens selection
Determine what the app is and therefore which lenses apply, then state the scope explicitly. Not every lens fits every app (a static marketing site needs no scaling or analytics pass; an internal tool needs no SEO). Selecting the wrong lenses pads the report with irrelevant findings and dilutes the real ones. The stack profile from Stage 0 informs this directly. `references/scope-and-lens-selection.md` covers reading the app, the lens-applicability map, and stating scope and what was excluded and why.

### Stage 2: Run the lenses (the passes)
Run each selected lens over the codebase, in priority order, each producing findings cited to file and line in the canonical schema (`references/finding-schema.md`), appended to the `raw-findings.jsonl` ledger as they're found — never summarised on the way in. Atomic lenses run first; the two **synthesis lenses** (soc2-compliance, adversary-emulation) run last and consume the ledger. The lenses and what each owns:
- **`code-audit`**: security, correctness, debugging, tests, structure, UI/UX (itself six deep passes; the security/correctness core).
- **`ai-saas-security`**: AI-specific abuse, prompt injection, cost runaway, token/rate limits (for apps with an AI surface).
- **`scaling-audit`**: data durability/backups, single points of failure, concurrency, dependency resilience, cost at scale, observability.
- **`release-and-ops`**: secrets, config/environments, rollback, CI/CD, reproducibility, release safety.
- **`data-privacy`**: data inventory/minimisation, consent/tracking, transparency, deletion/access rights, retention, breach readiness.
- **`performance`**: perceived speed, assets, bundle, rendering path, caching, runtime.
- **`accessibility`**: semantic structure, keyboard/focus, screen readers, colour/contrast, forms, testing.
- **`email-deliverability`**: authentication (SPF/DKIM/DMARC), reputation, list hygiene, content, transactional-vs-marketing.
- **`frontend-robustness`**: the four async states, form submission/double-submit, validation, slow/failed requests, optimistic updates/races, defensive rendering.
- **`internationalisation`**: text externalisation, locale formatting, timezones, RTL, Unicode (for apps targeting multiple locales).
- **`seo-discoverability`**: crawlability/rendering, indexing/canonical, metadata, social sharing, structured data, sitemaps (for public-facing apps).
- **`mobile-and-responsive`**: responsive layout, touch, viewport, mobile input, mobile performance, cross-device testing.
- **`analytics-and-instrumentation`**: meaningful metrics, event design, funnels, activation/retention, privacy-respecting tracking.
- **`anti-slop-writing`**: quality of the app's user-facing copy (landing/onboarding/microcopy/labels/empty-and-error wording/customer emails) — audits for AI-slop writing tells. The copy twin of frontend-design's visual audit; findings carry `category: content`, capped at medium, reported in the "Design & copy quality" section (for apps with real user-facing copy).
- **`code-quality`**: senior-review checklist — magic numbers, loose equality, empty catches, naming inconsistencies, async anti-patterns, boundary hygiene, mutation bugs, deferred implementations, and AI-generated code tells. Findings carry `category: code-quality`.
- **`dependency-audit`**: supply chain security — lock file hygiene, version pinning, dependency bloat, postinstall scripts, dev/prod boundaries, CI vulnerability scanning config. Reads lock files and manifests, never calls a registry. Findings carry `category: supply-chain`.
- **`infrastructure-config`**: deployment configuration files in the repo — Dockerfiles, Terraform, Kubernetes manifests, CI/CD pipelines, reverse proxy configs, secrets/env files. Checks for containers as root, open IAM, unpinned images, secrets in build args. Findings carry `category: infrastructure`.

Then the synthesis lens that runs within Stage 2 (reading the ledger of atomic findings):
- **`soc2-compliance`**: maps the technical-control subset of the Trust Services Criteria (CC1–CC9 + optional categories) onto code evidence, flags control gaps. Honest about the organisational-controls boundary. **Two triggers:** SOC 2 *certification readiness* when it is a stated goal (deferrable otherwise); **and** a mandatory reduced *data-protection-duty* pass (an impact assessment, a heightened lawful basis, children's-data duties — under whichever framework governs the data: GDPR/UK GDPR, HIPAA, PCI-DSS, COPPA, CCPA/CPRA…, confirmed by jurisdiction) whenever a regulated `data_classes` value is detected — that duty is in scope by data class, not by stated goal, and the harness enforces it.

**`adversary-emulation` runs after Stage 4** (not during Stage 2), consuming only the **reconciled, verified** finding set — see Stage 4. Chains must never be built on findings the verifier later refuted or the merge stage absorbed; the harness enforces this ordering by hard-failing any chain whose `component_findings` id is absent from the reconciled reported set.
- **`adversary-emulation`**: chains atomic findings into realistic attack paths (MITRE ATT&CK), assesses detection coverage, prescribes the cheapest break. Finds chained exploits lens-by-lens auditing misses. Strictly defensive (owned-system threat modelling). Run for any security-sensitive app.

`references/running-the-lenses.md` covers running each lens consistently, adapting it to the detected stack, the citation standard (file + line), the canonical schema, the ledger append, and the post-Stage-2 roll-call that confirms every selected lens actually ran.

### Stage 3: Merge and deduplicate
The same underlying issue often surfaces through several lenses (an open email relay is both a security and a deliverability finding; incomplete account deletion is both a correctness and a data-privacy finding; a client-rendered app that crawlers see as blank is both SEO and performance). Merge these into one finding that names all the lenses it touches, rather than reporting it several times. `references/merge-and-deduplicate.md` covers identifying cross-lens duplicates, merging them, and the common overlap map.

### Stage 4: Adversarial verification, severity calibration, and chain synthesis
Re-examine every critical and high finding against the actual code, as an adversary trying to disprove it: is it real, is it reachable, is the severity right? Record a structured verdict per finding (status + evidence quote) in its schema `verification` block; severity cannot exceed what the verdict supports, and a critical/high left unverified is capped at medium or it fails the harness. This drops false positives and corrects over- and under-rated severities, the step that makes the report trustworthy rather than alarmist. `references/verification-and-severity.md` covers the verification method, the structured verdict, the reasoning-finding gate, the severity scale, and recording when the verifier disagrees with the original lens.

**After reconciliation is complete** (merge + verification dispositions balanced, finding set frozen), run **`adversary-emulation`** against the reconciled, verified findings — never the raw pre-reconciliation ledger. Chains are then verified themselves (every critical/high chain through the same adversarial bar). A chain whose `component_findings` references a dropped or merged-away id is a dangling reference; the harness hard-fails it with a diagnostic naming whether to re-synthesise (refute-orphan) or rewrite (merge-orphan). State `chain.severity_basis` — the constituent ids the chain's severity rests on — so severity recomputation after constituent changes is mechanical and auditable.

### Stage 5: Consolidate and report — gated by the harness
Produce one report in the standard format: an executive summary that **leads with the gating set** (the verified critical/high count, not the inflated raw total — the reasoning mediums/lows are a backlog, stated with their confidence composition), the handful to fix first, findings by severity tier counting **distinct defects** (re-narrated attack chains fold into the root-cause finding they re-tell, never a second critical), each cited and verified with the fix and owning lens, a remediation order sequenced by **damage-rate-per-effort** (`live blast-radius × likelihood ÷ fix cost`, not severity label or data-sensitivity), a coverage matrix with the reconciliation line, and an explicit "what could not be assessed" section. Two scripts close the stage: **(1)** `node .claude/skills/production-audit/scripts/audit-check.mjs <audit-dir>` must exit zero — the gate that confirms every critical/high is verified-with-evidence or capped, no raw finding was lost, and the counts reconcile; **(2)** `node .claude/skills/production-audit/scripts/render-report.mjs <audit-dir> > report.md` generates the human-readable report *from* the validated `report.json`, so the method-line counts the reader trusts are computed from the gated data and cannot diverge. The report is not delivered until the harness exits zero. `references/report-format.md` and `references/coverage-matrix.md` cover the exact structure, the fix-first selection, the remediation ordering logic, and the honesty sections.

## Scoping the audit itself
Match depth and breadth to the app and the stakes. A small prototype gets the relevant lenses run lightly with an honest "you're fine for now, these few things matter before real users." A real app handling money, personal data, or real traffic gets the full applicable set at depth. Never pad: if a lens does not apply, say so in the scope section rather than manufacturing findings to look thorough. The goal is the few things that genuinely gate readiness, surfaced clearly, not the longest possible list.

## Skills this coordinates
All audit lenses above. For the fix phase (if chosen), it hands off to the craft skills under their own disciplines: `debugging-methodology`, `data-modelling`, `error-handling-patterns`, `api-and-interface-design`, `state-management`, `refactoring`, and `testing-strategy` (the test net that must exist before fixing safety- or money-critical paths). `anti-slop-writing` governs the report's prose.
